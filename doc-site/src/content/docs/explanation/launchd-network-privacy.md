---
title: launchd, LC_UUID, and the Tart bridge
description: Why elastic-fruit-runner needs a build-id-derived Mach-O UUID to reach Tart VMs on macOS 15+.
---

This page explains a non-obvious failure mode that affected `brew services`-launched daemons on macOS 15 (Sequoia) and how the runner is built to avoid it.

## The symptom

A daemon launched via `brew services start elastic-fruit-runner` (i.e. a per-user LaunchAgent) repeatedly fails to reach Tart VMs:

```text
waiting for SSH name=<vm> ip=192.168.64.X retry_in=16s
start runner failed err="... no route to host"
```

while at the very same moment, an interactive Terminal on the same host can SSH into the same VM with the same credentials.

## What is actually broken

Three independent pieces conspire here:

1. **Tart networks VMs through `bridge101`** (`192.168.64.0/24` by default). Reaching a VM IP requires the kernel to route via that interface.
2. **macOS 15 introduced [Local Network Privacy](https://developer.apple.com/documentation/technotes/tn3179-understanding-local-network-privacy)** for processes running under launchd as LaunchAgents. The kernel's network privacy machinery (nehelper) identifies the binary asking to dial a private subnet by its Mach-O `LC_UUID` load command.
3. **The Go internal linker historically did not emit `LC_UUID`** ([golang/go#68678](https://github.com/golang/go/issues/68678)). A binary without `LC_UUID` cannot be matched to any user-granted local-network permission, so the request gets rejected as `connect: no route to host` from the launchd context — even though the equivalent shell session works fine.

The Apple-signed `ssh` binary is exempt because it has its own `LC_UUID` and the system grants it access by default. That is why PR #71's `sshpass ssh ... true` workaround for SSH readiness is enough to pass CI even though the underlying Go process is still partially network-blocked.

## Why this is fixed in elastic-fruit-runner

Both the local `make build` target and the goreleaser config link with:

```text
-ldflags="-B gobuildid"
```

This tells the Go linker to derive the Mach-O `LC_UUID` from the (deterministic) Go build ID. The flag works on all Go versions ≥ 1.23 and is the same fix [cirruslabs/orchard#221](https://github.com/cirruslabs/orchard/issues/221) applied for the same problem.

You can verify a build:

```sh
otool -l "$(brew --prefix)/opt/elastic-fruit-runner/bin/elastic-fruit-runner" | grep -A1 LC_UUID
# cmd LC_UUID
# cmdsize 24
```

## Why we still keep the SSH-binary readiness probe

`waitForSSH` shells out to `sshpass`/`ssh` rather than dialing TCP/22 directly, even with the LC_UUID fix in place. Reasons:

- It mirrors the transport actually used by `Exec`, so a passing readiness probe guarantees the next step will work.
- It is robust against stripped/repacked binaries where `LC_UUID` may be missing (e.g. Homebrew bottles built without the ldflag).
- The same probe loop captures real SSH failures (auth, host-key churn during VM boot) that a TCP probe would silently classify as success.

On every probe loop the runner now also logs the kernel's chosen route to the VM (`route -n get <ip>`) so if a future regression resurfaces the same class of bug, the failure mode is obvious from logs.

## Fallback: run as a system LaunchDaemon

LaunchDaemons (loaded into `/Library/LaunchDaemons/` and run as root) are exempt from Local Network Privacy entirely. If you cannot upgrade for some reason, you can switch from the default per-user agent to a daemon by stopping the user-mode service and starting it under sudo:

```sh
brew services stop elastic-fruit-runner
sudo brew services start elastic-fruit-runner
```

The trade-off is that the runner now executes Tart and runner workloads as root, which most users will not want long-term. The LC_UUID build fix keeps the per-user, non-root LaunchAgent path working without elevated privileges.
