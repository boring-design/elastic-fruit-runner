---
title: "Why our launchd daemon couldn't reach Tart VMs (and Apple's ssh could)"
description: A debugging diary — from "no route to host" to a missing Mach-O LC_UUID load command.
date: 2026-04-26
---

> **TL;DR** — A `brew services`-launched daemon can hit `connect: no route to host` when dialing Tart VMs on macOS 15+, while a Terminal session on the same host has no problem reaching the same IP with the same credentials. The cause is not networking, sandboxing, or TCC. It's that older Go toolchains emit Mach-O binaries without an `LC_UUID` load command, and macOS 15's Local Network Privacy refuses to authorize a binary it can't identify. Fix: build with `-ldflags="-B gobuildid"`.

## The bug

We had been running `elastic-fruit-runner` on a Mac mini under `brew services` for a while. Then PR #71 lit up CI with this from the Tart backend:

```text
preparing runner runnerSet=fruit-runner-macos-arm64 runner=…d9024
cloning VM image=ghcr.io/cirruslabs/macos-tahoe-xcode:latest
starting VM name=fruit-runner-macos-arm64-d9024
waiting for VM IP name=fruit-runner-macos-arm64-d9024
waiting for SSH name=…d9024 ip=192.168.64.3 retry_in=1s
…
start runner failed err="SSH not reachable on …d9024 (192.168.64.3:22) after 2m0s"
stopping VM name=…d9024
deleting VM name=…d9024
```

Two minutes of TCP probes, every probe timing out, then the runner gave up and recycled the VM. Repeat forever.

The strange part: the same host could SSH into the same VM, with the same credentials, in the same window the daemon was failing:

```sh
sshpass -p admin ssh -o StrictHostKeyChecking=no admin@192.168.64.3 \
  "echo ok && sw_vers -productVersion"
# ok
# 26.3
```

So the network was reachable. The credentials were valid. But Go's `net.Dialer` in the daemon couldn't get a TCP handshake through, while OpenSSH from the shell could.

## The first hypothesis (wrong)

Sampling the live daemon showed every goroutine parked in `kevent` / `read` — no CPU pressure, no internal hot loop. The process was simply waiting on something external.

The Tart bridge is `bridge101`, `192.168.64.0/24`. A `route -n get 192.168.64.3` correctly named `bridge101` as the egress interface. So the kernel's routing table looked right.

But a `tcpdump` during the failure window caught this oddity:

```text
IP 192.168.1.87.<ephemeral> > 192.168.64.3.22: Flags [S]
```

That's the host's *physical LAN* source address dialing the VM, instead of the bridge interface's address (`192.168.64.1`). The SYN was going out, but on the wrong interface. We assumed something was redirecting the connection — a sandbox profile, a firewall rule, an entitlement check.

So we burned an afternoon on:

- TCC databases: no relevant entries
- LaunchAgent sandbox profiles: nothing applied to our daemon
- `pfctl` rules: empty
- `socketfilterfw` (Application Firewall): allowed
- Trying `Local Network Usage Description` plist keys: no effect

None of it explained why `ssh` worked and `net.Dialer` didn't. They run from the same launchd context, with the same environment, on the same host.

## The bypass

PR #71 unblocked CI by changing the readiness probe from a raw TCP dial to `sshpass ssh … true`. That worked, because Apple's `ssh` is a code-signed system binary — it has whatever entitlements / identity it needs, and the kernel authorized the dial.

The bypass shipped, the macOS integration job went green, and we filed [#73](https://github.com/boring-design/elastic-fruit-runner/issues/73) with a note: *"this is a workaround, not a fix — any future feature using a raw TCP socket from this daemon will fail again."*

## The actual cause

The fix was hiding in plain sight in two unrelated issues we hadn't connected:

- [golang/go#68678](https://github.com/golang/go/issues/68678) — *"cmd/link: LC_UUID not generated by go linker"*. Go's internal linker, before Go 1.24, did not emit the Mach-O `LC_UUID` load command unless you opted in. External linker (cgo enabled, `-buildmode=external`) did. Pure-Go static binaries didn't.
- [cirruslabs/orchard#221](https://github.com/cirruslabs/orchard/issues/221) — same project space (Tart, Cirrus's macOS VM ecosystem), same symptom, same fix. Their conclusion: macOS 15 (Sequoia) extended Local Network Privacy to launchd-spawned processes, and the kernel's `nehelper` identifies the binary asking to dial a private subnet by its `LC_UUID`. No `LC_UUID` → no identity → no permission match → connection refused.

That's it. That's the whole story:

1. **Tart routes VMs through `bridge101`** (`192.168.64.0/24`).
2. **macOS 15 wants to authorize anything talking to that subnet** via [Local Network Privacy](https://developer.apple.com/documentation/technotes/tn3179-understanding-local-network-privacy). The user grant is keyed by binary identity.
3. **macOS uses Mach-O `LC_UUID` as the identity**.
4. **The Go linker we built with didn't emit `LC_UUID`** → unidentifiable binary → request rejected → `connect: no route to host`.
5. **Apple-signed `ssh` has its own `LC_UUID`** and is granted by default → `ssh` works while our daemon fails, on the same socket address, in the same launchd context, at the same instant.

## The fix

A single linker flag. Both `make build` and the goreleaser config now link with:

```text
-ldflags="-B gobuildid"
```

This tells the Go linker to derive the Mach-O `LC_UUID` from the (deterministic) Go build ID. Works on Go ≥ 1.23, becomes the default in Go 1.24. After rebuilding:

```sh
otool -l output/elastic-fruit-runner | grep -A1 LC_UUID
# cmd LC_UUID
# cmdsize 24
```

A binary with `LC_UUID` is one the kernel can identify. Once identified, the existing user grant for the parent app applies, and `net.Dialer` to `192.168.64.3:22` succeeds from launchd just like it does from the shell.

## What we kept anyway

`waitForSSH` still shells out to `sshpass ssh … true` instead of dialing TCP/22 directly, even with `LC_UUID` in place. Three reasons:

- It mirrors the transport actually used by the next step (`Exec`). A passing readiness probe now guarantees the next operation will work, not just that a port is open.
- It's robust against repacked binaries that may have lost `LC_UUID` (Homebrew bottles built without the ldflag, third-party rebuilds, etc.).
- It catches real SSH failures — auth churn during VM boot, host-key changes — that a TCP probe would silently classify as "ready".

The probe loop now also logs the kernel's chosen route to the VM (`route -n get <ip>`) on probe start, and embeds the route into the timeout error. If this class of bug ever resurfaces, the failure mode is obvious from logs alone instead of requiring a `tcpdump`.

## Lessons

- **A working `ssh` next to a failing `net.Dial` is a binary-identity problem, not a network problem.** `ssh` is signed, your Go binary may not be.
- **Local Network Privacy applies to launchd-spawned processes on macOS 15+.** Anything dialing private subnets needs an identifiable binary.
- **`LC_UUID` is part of the binary's identity to the kernel.** Without it, the OS literally can't apply user permissions to your process.
- **Build flags are part of correctness.** A missing linker flag silently changed observable runtime behaviour.
- **`tcpdump` lied to us in a useful way.** The "wrong source address" SYN was a symptom of the kernel rejecting the bind, not of an interface misconfiguration. We chased the symptom for hours.

## Coda: making the grant stick across upgrades

The `LC_UUID` fix gets the connection through the kernel, but it has a UX consequence we initially shrugged at: `gobuildid` is *content-addressed*, so every release of `elastic-fruit-runner` produces a different UUID. macOS sees a different UUID as a different program, so the user gets re-prompted for Local Network Privacy on every upgrade. One click, but annoying enough to ask: is there a way for the grant to persist?

TN3179 has the answer: launchd plists can carry an `AssociatedBundleIdentifiers` key, and macOS will use the bundle id rather than `LC_UUID` as the identity for permission lookups. A bundle id is a stable string we choose; it doesn't change when source code does. Grant once, never re-prompt.

We hit a wall at the next layer: **Homebrew's `service do … end` DSL doesn't support `AssociatedBundleIdentifiers`.** The keys it emits into the plist are hardcoded in `Homebrew::Service#to_plist` (`Label`, `ProgramArguments`, `WorkingDirectory`, `KeepAlive`, …) with no extension point. `brew services start` regenerates the plist from the formula on every invocation, so post-install patching gets clobbered.

We almost went down a much heavier path here — a custom `elastic-fruit-runner launchd install` subcommand that managed its own plist out of band of `brew services`. We started building it. Then we re-read the brew services source.

The thing we missed first time: `Homebrew::Services::Cli.install_service_file` (`/opt/homebrew/Library/Homebrew/services/cli.rb:383`) treats `service.service_file.exist?` as authoritative. If a file at `<keg>/<plist_name>.plist` already exists, brew services *reads it verbatim* and copies it to `~/Library/LaunchAgents/`. The DSL's `to_plist` is never called. The DSL is only the fallback for formulas that don't ship their own plist.

So the recipe is simple: have the formula ship a plist into the keg with whatever keys we want.

```ruby
install do
  bin.install "elastic-fruit-runner"
  (prefix/"design.boringboring.elastic-fruit-runner.plist").write <<~EOS
    <?xml version="1.0" encoding="UTF-8"?>
    <!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "...">
    <plist version="1.0">
    <dict>
      <key>Label</key>
      <string>design.boringboring.elastic-fruit-runner</string>
      ...
      <key>AssociatedBundleIdentifiers</key>
      <array>
        <string>design.boringboring.elastic-fruit-runner</string>
      </array>
    </dict>
    </plist>
  EOS
end

service do
  name macos: "design.boringboring.elastic-fruit-runner"
  run [opt_bin/"elastic-fruit-runner"]
  keep_alive true
  ...
end
```

`name macos:` makes brew look for `<keg>/design.boringboring.elastic-fruit-runner.plist` (instead of the default `homebrew.mxcl.elastic-fruit-runner.plist`). Our `install` block writes exactly that file. brew services finds it, copies it to `~/Library/LaunchAgents/`, and `launchctl bootstrap`s it. End-to-end.

The user-facing flow is unchanged from before: `brew services start elastic-fruit-runner`. No new commands, no migration. The grant prompt fires once on first run; subsequent `brew upgrade`s do not re-prompt because the bundle id is stable, even though `LC_UUID` continues to change per build.

A few alternatives we considered and rejected:

- **Pinning `LC_UUID` to a fixed value.** Stable across builds, but breaks crash symbolication and `dSYM` matching, and Quinn warns that two programs sharing a UUID confuses NECP. The whole point of `gobuildid` is content-addressed identity — pinning it manually defeats that.
- **Patching Homebrew upstream.** The right long-term fix; we may still file an issue. But it's months away through review, merge, release, user upgrade. The shipped-plist trick works today and stays working even if upstream eventually adds the DSL key.
- **`brew services start --file=<our plist>`.** Functionally equivalent, but every user has to remember the `--file=` flag every time. Onboarding becomes "start the service with this 100-character command" — not great. Shipping the file inside the keg lets the regular `brew services start` pick it up.
- **A custom subcommand managing its own plist.** ~1100 lines of Go we'd own forever, plus a non-standard service-management UX. We almost shipped this; we're glad we didn't.

## Fallback: run as a LaunchDaemon

LaunchDaemons (loaded into `/Library/LaunchDaemons/`, run as root) are exempt from Local Network Privacy entirely. If you can't upgrade for some reason, switch from the per-user agent to a system daemon:

```sh
brew services stop elastic-fruit-runner
sudo brew services start elastic-fruit-runner
```

Trade-off: the runner now executes Tart and runner workloads as root. Most users won't want that long-term. The `LC_UUID` build fix keeps the per-user, non-root path working without elevated privileges, and is the recommended path.
